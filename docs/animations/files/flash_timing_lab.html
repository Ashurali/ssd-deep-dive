The flash timing & parallelism lab

Firmware performance work is reasoning about these timelines. Four scenes, from one shared bus to BiCS-style independent planes.

One channel = one shared 8-bit bus. CE# picks which die is listening; CLE/ALE flags say whether the bytes on the pins are a command, an address, or data. Eight pins carry everything — that's why channel time is precious and long media operations should never hog it. (Fig 2-8, 3-31→3-35)

Reading 4 pages with the book's own numbers (media→register 50 µs read + register→controller 50 µs transfer, drawn to scale). With the cache register, page N+1 loads from media while page N streams out — the transfers hide under the loads. Toggle it off and watch the timeline stretch. (Fig 3-8, §3.1.3's arithmetic)

Programming two pages (1.5 ms program, 50 µs data-in each). Single-plane: strictly serial. Dual-plane: one command programs both planes at once — 2 pages ≈ 1.5 ms + 2×50 µs instead of 2×(1.5 ms + 50 µs). Nearly double throughput, same die. (Figs 3-36→3-38 plane bits)

Classic lockstep: the die is one worker — the second read queues (tail-latency spike). AIPR: each plane is its own worker — concurrent reads, if they hit different planes. Same-plane collisions still serialize, which is why placement (striping hot data) matters. This is the 2nd edition's §5.4.3 — and BiCS8's read-QoS story — drawn as the figure the book never had.
